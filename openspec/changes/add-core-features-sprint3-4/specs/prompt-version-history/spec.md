# 提示词版本历史规范

## ADDED Requirements

### Requirement: 版本列表展示
系统SHALL提供提示词的完整版本历史列表,让用户能够追溯提示词的演变过程。

#### Scenario: 查看版本列表
- **WHEN** 用户在提示词详情页点击"版本历史"按钮
- **THEN** 系统展示该提示词的所有历史版本列表
- **AND** 每个版本显示版本号、修改时间、Token 数、变更说明(如果有)

#### Scenario: 版本列表为空
- **WHEN** 提示词是新创建的,只有一个初始版本
- **THEN** 系统显示仅有初始版本的提示信息
- **AND** 提示用户修改提示词内容后会自动创建新版本

### Requirement: 版本详情查看
系统SHALL允许用户查看任意历史版本的完整内容。

#### Scenario: 查看历史版本内容
- **WHEN** 用户点击版本列表中的某个历史版本
- **THEN** 系统在对话框或侧边栏中展示该版本的完整内容
- **AND** 显示该版本的详细元数据(版本号、Token 数、创建时间、变更说明)

#### Scenario: 对比当前版本
- **WHEN** 用户查看历史版本时
- **THEN** 系统提供"与当前版本对比"的选项
- **AND** 高亮显示内容差异(可选的高级功能)

### Requirement: 版本回滚
系统SHALL支持将提示词恢复到任意历史版本。

#### Scenario: 回滚到历史版本
- **WHEN** 用户在版本详情中点击"恢复此版本"按钮
- **THEN** 系统弹出确认对话框,说明回滚操作会创建新版本
- **AND** 用户确认后,系统将该历史版本的内容复制为最新版本
- **AND** 自动计算新版本的 Token 数
- **AND** 在变更说明中注明"恢复到版本 X"

#### Scenario: 回滚操作可追溯
- **WHEN** 用户执行版本回滚操作
- **THEN** 该操作本身作为一个新版本记录在历史中
- **AND** 版本历史保持完整,不删除任何历史版本

### Requirement: 版本创建规则
系统SHALL在提示词内容变更时自动创建新版本。

#### Scenario: 内容修改创建新版本
- **WHEN** 用户修改提示词的 content 字段并保存
- **THEN** 系统自动创建一个新的版本记录
- **AND** 版本号自动递增
- **AND** 记录修改时间和 Token 数

#### Scenario: 仅修改标签不创建版本
- **WHEN** 用户仅修改提示词的标签,不改变内容
- **THEN** 系统更新提示词的标签关联
- **BUT** 不创建新的版本记录

#### Scenario: 提供变更说明
- **WHEN** 用户在更新提示词时提供 change_note 参数
- **THEN** 系统将变更说明保存在新版本记录中
- **AND** 在版本列表中显示该说明

### Requirement: API 接口定义
系统SHALL提供标准的 RESTful API 支持版本历史功能。

#### Scenario: 获取版本列表
- **WHEN** 客户端发送 GET /api/prompts/{id}/versions 请求
- **THEN** 系统返回该提示词的所有版本列表
- **AND** 按版本号降序排序(最新版本在前)
- **AND** 包含分页参数支持(可选)

#### Scenario: 获取版本详情
- **WHEN** 客户端发送 GET /api/prompts/{id}/versions/{version_id} 请求
- **THEN** 系统返回该版本的完整信息
- **AND** 包括版本号、内容、Token 数、创建时间、变更说明

#### Scenario: 回滚到历史版本
- **WHEN** 客户端发送 POST /api/prompts/{id}/versions/{version_id}/restore 请求
- **THEN** 系统创建一个新版本,内容为指定历史版本的内容
- **AND** 返回新创建的版本信息和更新后的提示词信息
